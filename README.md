# MusicGuard (Go)

بوت حماية/نسخ لسيرفر ديسكورد مبني بـ Go، بأوامر **Prefix** وبصلاحيات محصورة على **Owners** من ملف الكونفق.

## الأوامر (Prefix)
- `!protect`: يلتقط نسخة مرجعية كاملة **فورًا** (مرة واحدة) ويثبتها كأساس للحماية.
- `!backup`: يحمّل النسخة المرجعية ويطبق الفروقات على السيرفر بدون حذف شامل.
- `!protect-user`: يرسل لوحة Embedded فيها 3 أزرار (إضافة موثوق / إزالة موثوق / تحديث).
- `!trust <userID>`: إضافة موثوق.
- `!untrust <userID>`: إزالة موثوق.

> كل الأوامر يعملها فقط أصحاب المعرفات داخل `owner_id` و `owner_ids` في `config.json`.

## الأربع نقاط المنفذة للأداء الأعلى
1. **In-memory baseline + WAL**
   - الحالة المرجعية تبقى بالذاكرة.
   - يتم تسجيل كل حدث مهم في `data/events.log` (WAL) بدل الاعتماد على إعادة قراءة كاملة دائمًا.
2. **Delta Reconcile (ID-first)**
   - عند كل حدث، النظام يحدد الـ targets المتأثرة (رول/روم) ويصلحها فقط.
   - المطابقة تكون أولاً بالـID ثم fallback بالاسم لتقليل الإنشاءات غير الضرورية.
3. **Priority lanes**
   - المسارات تعمل بالتوازي: أمني (عقاب/سحب صلاحيات) + استعادة + طوارئ (drift).
4. **Resource-key locks**
   - أقفال حسب المفتاح (`role:<id>` / `channel:<id>`) لتفادي تعارضات الكتابة على نفس المورد مع إبقاء التوازي عالي.

## منطق التحديث التلقائي للنسخة
- إذا الفاعل **موثوق** وعدّل شيء، يتم تحديث baseline فورًا (Capture + Save) حتى تصبح تعديلاته هي النسخة الجديدة المعتمدة.

## التشغيل
1. عدّل `config.json`.
2. شغّل:
   ```bash
   go mod tidy
   go run ./cmd/bot
   ```

## ملاحظة واقعية
- أقصى سرعة فعلية تبقى محدودة بـ Discord API Rate Limits وهرمية الصلاحيات، حتى مع أعلى توازي داخل Go.


## دقة الكاتوقري/الأب-الابن
- الاستعادة تعمل بمرحلتين: إنشاء/تحديث الكاتوقريات أولاً، ثم القنوات التابعة.
- يتم عمل remap لـ ParentID (من ID القديم إلى ID الحالي) لتفادي فقدان الهيكل.
- إذا القناة موجودة بالفعل (ID أو اسم)، يتم نقلها/تحديثها فوراً بدل إنشاء قناة جديدة.
- **جميع Permission Overwrites** للرومات/الكاتوقري يتم حفظها واستعادتها (Allow/Deny + نوع الهدف role/member).

## ملاحظة عن Rate Limit وTimeout
- تم إزالة قيد timeout المحلي (8 ثواني) من مسار الاستجابة للأحداث.
- تم إزالة قيد التوازي الداخلي المعتمد على semaphore داخل الاستعادة/إزالة Admin، ولم يعد هناك اعتماد على رقم worker_count.
- **لكن لا يمكن إزالة Rate Limit الخاص بـ Discord API** لأنه قيد على مستوى خوادم Discord نفسها.
